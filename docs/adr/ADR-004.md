# ADR-004: Estrategia de seguridad

## Contexto
El análisis del legacy identificó un sistema con vulnerabilidades
críticas en producción. Los usuarios accedían al sistema diariamente
mientras estas vulnerabilidades existían sin corrección:

- SQL Injection en 13+ puntos: search_products, filter_by_warehouse,
  get_sales_by_user (duplicado en dos archivos), export_report,
  list_for_user, search_by_kind, latest_for_user, list_by_status,
  count_unread, search_refunds, list_refunds_by_user, download_csv.
  Varios accesibles sin autenticación.
- Auth bypass: require_admin(data) verificaba data.get('is_admin') —
  cualquier request con {"is_admin": true} en el body pasaba como admin
  sin validación contra ninguna sesión real.
- Contraseñas en texto plano: la tabla users almacenaba passwords
  sin hash. Una filtración exponía todas las credenciales directamente.
- Stock negativo posible: decrement_stock restaba sin verificar
  si el resultado sería negativo.
- Caché con invalidación incorrecta: _NOTIF_CACHE no se invalidaba
  correctamente en delete y create, retornando datos desactualizados.

## Opciones consideradas

**Parchear el legacy con correcciones puntuales**
Agregar parámetros a queries existentes, cambiar require_admin para
verificar contra sesión en base de datos, hashear contraseñas nuevas
sin migrar las existentes. Rápido pero propenso a omisiones — las
vulnerabilidades están dispersas en todo el código.

**Rediseñar con seguridad como restricción de diseño desde el inicio**
Tomar cada decisión arquitectónica con seguridad como requisito no
negociable. Queries parametrizadas como única forma posible de acceder
a datos, autenticación basada en tokens firmados imposibles de falsificar
desde el cliente, contraseñas hasheadas antes de tocar la base de datos.

## Decisión
Rediseñar con seguridad como restricción de diseño, aplicando estas
correcciones de forma sistemática en todo el sistema nuevo:

**SQL Injection → Prisma como única capa de acceso a datos**
Ningún módulo construye strings SQL. Todo acceso a datos pasa por el
ORM con parámetros tipados. El único lugar con where dinámico es
ExportRepositoryPrisma, construido programáticamente con el tipo
Prisma.SaleWhereInput — nunca concatenando strings.

**Auth bypass → JWT firmado con secret del servidor**
auth.middleware.ts implementa verifyToken que lee el header
Authorization: Bearer <token>, verifica la firma con JWT_SECRET
del entorno del servidor y extrae el payload { userId, username, isAdmin }.
Ningún dato del body determina permisos.

**Contraseñas en texto plano → bcrypt con salt rounds 10**
Login.ts usa bcrypt.compare() para verificar credenciales. El script
seed.ts hashea todas las contraseñas del legacy con bcrypt.hash()
antes de insertarlas. Ningún módulo almacena o compara contraseñas
en texto plano.

**Stock negativo → validación en dominio antes de persistir**
La entidad InventoryStock expone canDecrement(amount: number): boolean
que verifica this.quantity >= amount. AdjustStock llama este método
antes de cualquier decremento. Si retorna false lanza "Stock insuficiente"
antes de tocar la base de datos.

**Caché inconsistente → sin caché en memoria**
El sistema nuevo no implementa caché en memoria para notificaciones.
SQLite con Prisma es suficiente para la carga de un ERP de PyME.

## Consecuencias

**Positivas:**
- Las 13+ vulnerabilidades de SQL injection son estructuralmente
  imposibles — es una restricción de diseño, no una corrección puntual
- Un atacante no puede escalar privilegios modificando el body del request
- Una filtración de la base de datos no expone contraseñas en texto plano
- El stock negativo es imposible a nivel de dominio
- La eliminación del caché en memoria elimina bugs de estado compartido

**Negativas:**
- JWT sin revocación inmediata — un token comprometido sigue válido
  hasta su expiración. Mitigación futura: refresh tokens con lista
  de revocación o tokens de corta duración
- bcrypt en la migración es lento por diseño — hashear cientos de
  usuarios tarda varios segundos. Costo único aceptable
- Sin caché, cada petición de notificaciones hace una query a SQLite.
  Aceptable para el volumen actual; necesitará estrategia de caché
  si el sistema escala
