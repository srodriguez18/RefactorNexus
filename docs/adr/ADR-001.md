# ADR-001: Elección de arquitectura

## Contexto
Se recibió un monolito Python/Flask fuertemente acoplado sin separación
de capas. El archivo principal de rutas mezclaba parsing HTTP, validación,
acceso a datos, reglas de negocio y manejo de sesión en el mismo lugar.
La lógica de descuentos estaba duplicada en 3 archivos distintos:
logic/finance.py, logic/reports.py y logic/exports.py.
Había 13+ puntos de SQL injection, triggers SQL con lógica de negocio,
y contraseñas en texto plano. La dirección decidió modernizar el stack
completo a TypeScript fullstack.

## Opciones consideradas

**Clean Architecture con módulos por bounded context y shared kernel**
Separación explícita en capas: domain/, application/, infrastructure/,
interface/. Cada bounded context es un módulo independiente. La lógica
financiera se extrae a un shared kernel (packages/finance) importado
por los módulos que la necesitan.

**Vertical Slices**
Organización por feature completa, sin separación rígida de capas.
Más simple de entender pero con mayor riesgo de mezclar responsabilidades
cuando el sistema crece.

**MVC en capas tradicional**
Models, Controllers, Services en carpetas globales. Familiar pero tiende
a recrear el mismo acoplamiento del legacy cuando el sistema crece.

## Decisión
Clean Architecture con módulos por bounded context y packages/finance
como shared kernel.

## Consecuencias

**Positivas:**
- La lógica de negocio en domain/ y application/ es testeable sin
  levantar servidor ni base de datos
- El shared kernel packages/finance elimina la triplicación de lógica
  de descuentos que existía en el legacy
- Las fronteras entre módulos son explícitas — ningún módulo accede
  directamente a las tablas de otro
- La inversión de dependencias permite cambiar Prisma por otro ORM
  sin tocar los casos de uso
- El bug LEGACY_A se resolvió en la capa de dominio, no con un trigger SQL

**Negativas:**
- Mayor cantidad de archivos por módulo — cada funcionalidad requiere
  al menos 4 capas
- Curva de aprendizaje para desarrolladores acostumbrados a MVC
- Para funcionalidades simples puede sentirse sobreingeniería
