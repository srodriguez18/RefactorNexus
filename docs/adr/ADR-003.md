# ADR-003: Estrategia de migración de datos

## Contexto
El legacy tenía un archivo seed_data.sql declarado inmutable por las
reglas del proyecto. Este archivo contenía datos de producción con
múltiples problemas de calidad que no podían resolverse modificando
el archivo directamente:

- Fechas en 4 formatos distintos: ISO estándar, ISO con hora,
  DD/MM/YYYY y Unix timestamp en segundos
- Campo amount en refunds guardado como TEXT en lugar de número
  (ej: '3335.5' con comillas)
- Campo total en sales guardado como TEXT en lugar de número
- Status de refunds en 6 variantes del mismo valor:
  'pending', 'Approved', 'aprobada', 'done', 'rejected'
- Status de notifications en 4 variantes:
  'unread', 'read', 'READ', 'leido'
- Contraseñas en texto plano en la tabla users
- Foreign keys sin enforcement (PRAGMA foreign_keys = OFF) —
  algunos registros referencian IDs inexistentes

## Opciones consideradas

**Reusar el mismo archivo .db del legacy**
Apuntar DATABASE_URL al archivo data/legacy.db existente. Rápido
pero imposible — el schema nuevo tiene columnas distintas, tipos
corregidos y nombres diferentes. Prisma no puede usar una base de
datos cuyo schema no controla.

**Script ETL que lea el legacy.db y migre al nuevo schema**
Conectar dos bases de datos simultáneamente, leer de una e insertar
en la otra. Más complejo de mantener y requiere tener el legacy
corriendo en paralelo.

**Prisma migrations desde cero con script de seed normalizado**
Crear el schema nuevo con Prisma, correr prisma migrate dev, y
construir un script seed.ts que lea el seed_data.sql sin modificarlo,
normalice todos los datos en memoria y los inserte en la nueva base
de datos limpia.

## Decisión
Prisma migrations desde cero con script apps/api/prisma/seed.ts
que lee y normaliza seed_data.sql.

El script implementa:
- parseDate(): detecta y normaliza los 4 formatos de fecha
- parseRefundStatus(): normaliza las 6 variantes a
  pending | approved | rejected
- parseNotificationStatus(): normaliza las 4 variantes a
  unread | read
- Hashing de contraseñas con bcrypt durante la migración
- parseFloat() en campos TEXT numéricos
- Omisión con log de registros con foreign keys inválidas
- Idempotencia: verifica si ya hay datos antes de insertar

## Consecuencias

**Positivas:**
- seed_data.sql no se modifica — respeta la restricción del proyecto
- Los datos quedan normalizados desde el primer arranque: tipos
  correctos, estados consistentes, contraseñas hasheadas
- El schema nuevo es limpio y no hereda los tipos incorrectos del legacy
- El script es idempotente — se puede correr múltiples veces sin
  duplicar datos
- Prisma Studio permitió verificar visualmente que la migración
  fue correcta
- La normalización está documentada y es reproducible

**Negativas:**
- Requiere mantener el script de seed sincronizado si el schema cambia
- Los datos históricos de refunds con status 'done' o 'aprobada'
  se mapearon a 'approved' — se pierde el matiz semántico original
- El campo discount no existía en el legacy — las ventas históricas
  migradas tienen discount = 0 aunque en realidad tuvieran descuentos
  aplicados por el trigger SQL
